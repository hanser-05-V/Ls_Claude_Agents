# 设计方案：子 Agent 标签上下文系统

> 状态：🟢 已决议 — **当前使用 v2.0，v2.1 作为后续升级方向**
> 决议时间：2026-06-29
> 决议：帅老大确认先使用 v2.0 协作协议，v2.1 标签上下文池待 P0 风险解决后再落地

## 一、推荐方案（帅老大确认版）

| 维度 | 方案 |
|------|------|
| 标签粒度 | 回合级 `#T1-R1` `#T1-R2` |
| 写入方式 | 混合（主脑自动 + 帅老大手动） |
| 上下文池 | 任务级日志 `T{N}_上下文.md` 持续追加 |
| 子agent读取 | 控制器裁剪（当前回合全文 + 上一回合关键结论） |
| Compact | 每标签独立压缩：已完成回合→结构化摘要，活跃回合保留原文 |
| 帅老大编辑权限 | ✅ 可追加笔记 |
| 跨任务读取 | ❌ 只读自己的 |

## 二、各版本上下文丢失对比

```
┌──────────────────────────────────────────────────────────────────┐
│  场景：Bug修复（排查→编码→审查→修复→再审查）                       │
│  链长：3跳（Bug侦探 → 编码工匠 → 审查官）                         │
└──────────────────────────────────────────────────────────────────┘

v0（无协议 — 随便派）
  主脑: "帮我修个bug"  → 子agent不知道bug在哪、不知道上下文
  丢失: 60-70%  ← 基本靠猜

v1.0（交接文件 + 摘要）
  主脑看对话 → 写摘要 → 交接文件 → 子agent读摘要
  第1跳(Bug排查→编码): 主脑摘 Bug侦探结论: 20%丢失
  第2跳(编码→审查): 审查官读交接文件+代码: 15%丢失
  第3跳(审查→修复): 主脑摘要审查意见: 20%丢失
  累计: 40-50%丢失
  痛点: 每次摘要都是主脑"用自己的话重说一遍"，原话丢失

v2.0（自包含 Prompt + 两阶段审查）
  主脑精读源码 → 构造完整 prompt（贴原文代码/需求）
  第1跳: 编码工匠看到完整 spec + 源码片段: 10-15%丢失
  第2跳: Spec审查独立验证，不看摘要看代码: 5%丢失  
  第3跳: 代码审查独立验证: 5%丢失
  累计: ≤25%丢失
  改进: 主脑不摘要，贴原文。审查者自己看代码。

v2.1（标签上下文池 — 讨论中）
  所有讨论自动归档到 T1_上下文.md，子agent启动时读取裁剪后的原文
  第1跳: 编码工匠看到"帅老大原话+Bug侦探原报告": ≤8%丢失
  第2跳: Spec审查看到"需求原文+实施报告+D1代码diff": ≤5%丢失
  第3跳: 编码工匠修复时看到"上一轮的审查原文": ≤5%丢失
  累计: ≤15%丢失
  改进: 控制器从"翻译官"变成"索引员"，子agent看到的是原文而非转述

┌──────────────────────────────────────────────────────────────────┐
│  场景：简单单任务（设计 → 编码 → 审查，理想情况）                    │
└──────────────────────────────────────────────────────────────────┘

v0:  50-60%
v1.0: 25-35%
v2.0: 10-15%
v2.1: 5-8%

┌──────────────────────────────────────────────────────────────────┐
│  场景：复杂多轮修复（编码 → 审查 → 修复 → 审查 → 修复... x3轮）     │
└──────────────────────────────────────────────────────────────────┘

v0:  累积失真 80%+（每轮都是新prompt，不知道上次做了什么）
v1.0: 55-65%（每轮摘要叠加失真）
v2.0: 30-40%（每轮重新构造prompt，但看不到"为什么上次那样修"）
v2.1: 15-20%（上下文池保留了每轮的原文推理链路）
```

## 三、v2.1 标签上下文系统的缺陷与隐患

### 🔴 P0 — 可能造成严重问题

#### 1. 信息过时（旧决策污染当前判断）

```
T1_上下文.md:
  R1: 帅老大说"用同步加载"  ← 早期决策
  R2: 架构导师说"同步加载在这里不行，改异步"
  R3: 编码工匠在修 R3 时读到 R1 的"同步加载" → 按旧决策写了 ← 🔴 Bug!

根因: 上下文池是追加的，后来的修正和早期的决策混在一起
     子 agent 没有能力区分"哪个是最新的有效决策"
     
缓解: 裁剪时给每条信息打时间戳 + 状态标记 [有效]/[已废弃]
      但谁来标记"已废弃"？需要主脑或帅老大显式标注
```

#### 2. 上下文池膨胀失控

```
一个复杂功能可能：
  R1: 500行 → R2: 400行 → R3: 350行 → R4: 300行 → R5: 250行
  共 1800 行上下文

子 agent prompt 一般有效窗口 ~8-15k tokens
如果 1800 行中文 ≈ 5000 tokens → 还装得下
但如果 10 轮 + 源码粘贴 → 轻松爆窗口

Compact 缓解但非根治：
  压缩到结构化摘要：200行摘要 + 当前回合400行 = 600行
  丢失: 之前回合的过程细节（"帅老大为什么改了主意"这类）
```

#### 3. 标签归属模糊（跨任务交叉讨论）

```
实际场景：
帅老大: "#T1 这个登录页闪退…我觉得是网络回调的问题"
  → 追加到 T1_上下文.md ✅

帅老大: "等等，说到网络回调，#T2 那个支付页面也有类似问题"
  → 这句话归 T1 还是 T2？
  → 归 T1: T2 缺了这段上下文，编码工匠不知道支付页也要改
  → 归 T2: T1 断了上下文，前后讨论不连贯
  → 两个都归: 重复存储，子 agent 看到不该看的内容

根因: 真实对话不是线性的，一个话题经常跳到另一个
     标签系统假设"每个任务独立"，但实际不独立
```

### 🟡 P1 — 有解决方案但需注意

#### 4. 子 agent "自主读取"的幻觉

```
期望: 子 agent 读了上下文池 → 理解了完整背景 → 做出更好决策
现实风险: 子 agent 可能:
  a) 读太多 → 信息过载 → 忽略关键信息（和没读一样）
  b) 读太片面 → 只看到 R1 没看到 R2 的修正 → 误解
  c) 过度相信上下文 → 把帅老大的随口一说当铁律

Superpowers 的做法恰恰相反：
  控制器构造"刚好够用"的 prompt，不给子 agent 选择权
  因为"选择看什么"本身就是一个容易出错的任务
```

#### 5. 标签维护的心智负担（主脑侧）

```
主脑每轮都要：
  1. 判断帅老大这句话归哪个任务
  2. 判断归哪个回合
  3. 判断该追加还是该新建回合
  4. 写入上下文池
  5. 裁剪出子 agent 需要的部分

如果同时 3 个任务活跃，主脑要同时维护 3 个上下文池
→ 主脑自己的上下文被"元管理"消耗 → 主窗口扮演质量下降
```

#### 6. Compact 压缩丢失的隐性信息

```
R1 原文（200行）→ Compact → 结构化摘要（15行）

结构化摘要保留的：
  ✅ 原始需求：一行
  ✅ 关键决策：一行  
  ✅ 审查结论：一行

丢失的：
  ❌ "帅老大为什么改主意" —— 决策的推理过程
  ❌ "讨论过的备选方案" —— 下次遇到类似场景有价值
  ❌ "帅老大当时的语气/强调" —— 影响优先级判断
  ❌ "讨论中被否决的方案" —— 防止子 agent 重新踩坑

Compact 本质是压缩，压缩就一定有损。
如果子 agent 后续需要"为什么上次这么修"的推理过程，
而这个过程在 compact 中被丢了 → 问题
```

### 🟢 P2 — 小问题，可接受

#### 7. 目录结构膨胀

```
E:\LS_Claude_Agents\{会话}\
├── 上下文池\           ← 新增
│   ├── T1_上下文.md
│   ├── T2_上下文.md
│   └── T3_上下文.md
├── 交接记录\           ← 原有
├── agent_会话记录\     ← 原有
├── 任务墙.md
├── 协作协议.md
└── RESUME.md

上下文池 和 交接记录 功能有重叠——
交接记录也是"上游→下游"的信息传递
两套体系并存可能混乱
```

#### 8. 子 agent 并行时的写入冲突

```
两个子 agent 同时执行 T1 和 T2（非严格冲突但都涉及 T3 共享模块）
→ 都往 T3_上下文.md 写
→ 如果同时写 → 冲突

实际概率低（Fresh Agent 一次只做一件事）
但在并行派发 + 共享上下游时可能出现
```

## 四、风险评估总表

| 风险 | 级别 | 概率 | 影响 | 缓解手段 |
|------|:--:|:--:|:--:|------|
| 信息过时（旧决策污染） | 🔴P0 | 高 | 高 | 信息打时间戳+状态标记 |
| 上下文池膨胀 | 🔴P0 | 中 | 中 | Compact + 裁剪策略 |
| 标签归属模糊 | 🔴P0 | 高 | 中 | 允许一条信息归多个标签 |
| 子agent自主读取失败 | 🟡P1 | 中 | 中 | 主脑裁剪 > 子agent自主选择 |
| 主脑元管理消耗 | 🟡P1 | 中 | 低 | 简化标记规则 |
| Compact信息丢失 | 🟡P1 | 中 | 中 | 保留推理链摘要，不只存结论 |
| 目录结构重叠 | 🟢P2 | 低 | 低 | 合并上下文池和交接记录 |
| 并行写入冲突 | 🟢P2 | 极低 | 低 | 单任务单线程写入 |

## 五、我的判断

**v2.1 标签上下文池方案，收益最大的场景是「多次修复的复杂任务」**——它能保留推理链路，让子 agent 知道「上次为什么那样做、为什么被推翻」。这个场景 v2.0 做不好。

**但三个 P0 风险需要先解决才能落地：**

| P0 | 解决方向 |
|----|---------|
| 信息过时 | 每条上下文打 `[有效]`/`[已废弃]` 状态标记，子 agent 优先读 [有效] |
| 上下文池膨胀 | Compact 时保留「决策推理摘要」而不只是结论，限制每回合活跃行数 ≤ 200 行 |
| 标签归属模糊 | 允许多标签归属（一条讨论同时归 #T1 #T2），避免硬切分 |

**建议：不单独建上下文池目录，直接把交接文件升级为"持续追加 + 多回合"模式** —— 减少目录重叠，现有体系可以承载。
