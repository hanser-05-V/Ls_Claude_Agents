## 🧬 帅老大定制规则

### 核心哲学

- (2026-06-29) **协作协议 v2.0**：思考在主窗口，执行丢给子 agent。讨论/分析/设计/审查/教学 → 主窗口角色扮演（零丢失）。纯执行（写代码/查Bug/批量操作）→ 子 agent 自包含 prompt（有损但可控 ≤15%）

### 主窗口扮演

- (2026-06-29) **主窗口扮演规则**：主脑读取对应主管的 agent 配置文件（`~/.claude/agents/ls-*.md`），在主窗口切换角色口吻对话。规则是他们的，执行模型是主脑的，上下文零丢失。
- (2026-06-29) **@ 直接指派规则**：帅老大 @主管名 → 主脑立即读取该主管规则 → 切换角色扮演（不走子 agent）。只有帅老大明确说"派子 agent"或任务为纯执行时才启动子 agent。

### 子 Agent 派发（整合 Superpowers 范式）

- (2026-06-29) **自包含 Prompt 构造（P0 铁律）**：派子 agent 时，控制器必须先精读相关源码，构造**完整自包含 prompt**——包含完整任务描述 + 相关代码片段 + 设计约束 + 验收标准。子 agent 绝不继承主会话历史，只看到注入的 prompt。目标：丢失 ≤15%。
- (2026-06-29) **Fresh Agent Per Task**：每个独立任务启动一个全新子 agent，做完即关。不做"一个 agent 做多件事"。
- (2026-06-29) **子 Agent 模型选择**：机械执行任务（1-2文件/清晰spec）→ 便宜模型。多文件集成/模式匹配 → 标准模型。架构/设计/审查 → 最强模型。不一律用同一模型。
- (2026-06-29) **两阶段审查循环**：编码完成后——①Spec 审查子 agent（独立，验证是否符合需求，不信任实施者报告）→ 通过后 → ②代码质量审查子 agent（独立，只审代码质量）→ 通过后才标记任务完成。两个审查者都必须自己去读代码。
- (2026-06-29) **子 Agent 升级协议**：实施者返回 DONE → 进入审查。DONE_WITH_CONCERNS → 主脑读疑虑后决定。NEEDS_CONTEXT → 补充上下文重派。BLOCKED → 分析原因（缺上下文/能力不够/任务太大/计划有问题），选择升级模型或拆分任务或报告帅老大。**绝不忽略升级或将同一模型重试。**

### 部门交接

- (2026-06-29) **部门交接铁律**：每次部门间交接必须写入交接文件 `交接_{任务ID}_{上游角色}→{下游角色}.md`。下游开工前先读取上游交接文件 + 帅老大原始需求。禁止裸交接。

### 审查铁律

- (2026-06-29) **审查不信任原则**：审查者被明确告知"不要信任实施者的报告"。必须自己去读代码、自己验证。实施者说"已完成"不代表已完成。

## ⚡ 催化剂（注入各主管系统提示）

- **架构导师**：优先在主窗口扮演模式进行教学，仅在纯文件搜索时启动子 agent
- **编码工匠**：开工前先读取上游交接文件，理解帅老大原始意图再编码。收到审查者反馈后立即修复，修复后接受 re-review
- **品质检查官**：分为两个独立子 agent：①Spec 审查者（先审，对照需求逐条验证）→ ②代码质量审查者（后审，对标最佳实践）。审查时优先使用主窗口扮演模式。明确告知：不信任实施者的自述报告
- **文档书记**：所有交接记录和会话记录按协作协议目录结构归档。记忆库修改（除自动规则添加外）需帅老大确认
- **Bug侦探**：查 Bug 前读取交接文件中的原始需求描述。排查结果写入交接文件，供编码工匠开修前阅读
- **小爱助理**：监控各主管上下文丢失量，超过 15%（子 agent 模式）立即报告帅老大
- **主脑统筹**：派子 agent 前必须精读源码 + 交接文件，构造自包含 prompt。子 agent 返回后按升级协议处理。不跳过审查循环。不忽略子 agent 的升级信号

# 协作协议 v2.0（完整版）

> 整合了 Superpowers 的 `dispatching-parallel-agents` + `subagent-driven-development` 范式

---

## 核心原则

**思考在主窗口，执行丢给子 agent。**

---

## 四种协作模式

| 模式 | 场景 | 上下文丢失 | 方式 |
|------|------|:--:|------|
| **主窗口扮演** | 讨论/分析/设计/审查/教学 | **0%** | 主脑读取对应 agent 配置，切换角色口吻 |
| **子 Agent 自包含** | 写代码/查Bug/批量操作 | **≤15%** | 控制器精读源码→构造完整自包含 prompt→注入 |
| **两阶段审查** | Spec合规 + 代码质量验证 | 独立子agent各审各的 | ①Spec审查子agent → ②代码质量审查子agent |
| **链式交接** | 设计→编码→审查多部门 | 10-15% | 每环节留交接文件，下环节先读再开工 |

---

## 多部门协作信息流

```
                      帅老大
                        ↕ 零丢失（主窗口扮演）
                    🧠 主脑统筹
                    （精读源码 + 构造自包含 prompt）
                        │
          ┌─────────────┼─────────────┐
          ▼             ▼             ▼
     [子Agent·A]   [子Agent·B]   [子Agent·C]
     设计/排查      执行/编码      审查/验证
     (可能扮演)     (自包含)      (扮演或子Agent)
          │             │             │
          └──────┬──────┴──────┬──────┘
                 ▼             ▼
            交接文件       审查循环
         (上游→下游)    (spec→quality)
                 │             │
                 └──────┬──────┘
                        ▼
                   主脑汇总 → 帅老大
```

**铁律：**
- 子 agent 之间 ❌ 不直接通信，全部通过交接文件 + 主脑中转
- 链上环节 ≤ 3 跳（设计→编码→审查），避免多次摘要失真
- 控制器派子 agent 前**必须**自己先读过相关源码，再构造 prompt
- 审查者**不信任**实施者报告，独立验证

---

## 审查循环（整合 Superpowers）

```
实施者子 Agent 完成
    │
    ├── 状态 DONE → 进入审查
    ├── 状态 DONE_WITH_CONCERNS → 主脑读疑虑 → 决定
    ├── 状态 NEEDS_CONTEXT → 主脑补充 → 重派
    └── 状态 BLOCKED → 分析原因 → 升级模型/拆分任务/报告帅老大
         │
         ▼
  ┌──────────────────────┐
  │ ① Spec 审查子 Agent   │  ← 独立，对照需求逐条验证
  │   ❌ 不合格 → 实施者修复 → re-review
  │   ✅ 合格 → 进入下一步
  └──────────────────────┘
         │
         ▼
  ┌──────────────────────┐
  │ ② 代码质量审查子 Agent │  ← 独立，对标最佳实践
  │   ❌ 不合格 → 实施者修复 → re-review
  │   ✅ 合格 → 任务标记完成
  └──────────────────────┘
```

---

## 子 Agent Prompt 构造规范

控制器在派子 agent 前，必须构造包含以下要素的自包含 prompt：

```markdown
1. 完整任务描述（直接粘贴，不让子 agent 读文件）
2. 场景上下文（该任务在整个工作中的位置）
3. 相关源码片段（控制器已读过，粘贴关键代码）
4. 设计约束（命名规范、文件结构、不能改变的范围）
5. 验收标准（如何判断完成）
6. 上游交接文件（如有，粘贴核心内容）
7. 帅老大原始需求原话
```

## v2.1 — 子 Agent 标签上下文池（设计中）

**目标：** 让子 agent 读取主窗口原始讨论（而非主脑摘要），上下文丢失从 ≤15% 压缩到 ≤8%

**核心方案：**
- 标签编码：`#T{N}-R{M}`（回合级，如 `#T1-R2`）
- 写入：混合标记（主脑自动 + 帅老大手动）
- 上下文池：`交接记录/` 升级为持续追加日志，按任务编号分段
- 子 agent 读取：控制器裁剪（当前回合全文 + 上一回合关键结论，不摘要只裁剪）
- Compact：每标签独立压缩，已完成回合→结构化摘要，活跃回合保留原文

**版本对比：**

| 场景 | v1.0 | v2.0 | v2.1 |
|------|:--:|:--:|:--:|
| Bug修复 3跳链 | 40-50% | ≤25% | ≤15% |
| 简单单任务 | 25-35% | 10-15% | 5-8% |
| 多轮修复（N≥3轮） | 55-65% | 30-40% | 15-20% |

**待解决的 P0 风险：**

| 风险 | 级别 | 解决方向 |
|------|:--:|------|
| 信息过时污染（旧决策被读到） | 🔴P0 | 每条信息打 `[有效]`/`[已废弃]` 状态标记 |
| 上下文池膨胀（多轮后爆窗口） | 🔴P0 | Compact 保留推理链摘要，每回合活跃 ≤200行 |
| 标签归属模糊（跨任务交叉讨论） | 🔴P0 | 允许多标签归属，一条讨论同时归 #T1 #T2 |

**设计文档：** 参见各会话记录目录下的 `设计讨论_子Agent标签上下文系统.md`
