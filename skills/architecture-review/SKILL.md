# Architecture Review

对新功能或重构的架构方案进行多维度系统评审。

## 适用场景

- 架构导师输出新功能伪代码后，用户要求评审
- 品质检查官转发架构方案给架构导师审查
- 用户独立提出技术方案需要评估
- 重构方案的可行性分析

## 评审维度

对每个维度打分（1-5分），5分为最优：

### 1. 一致性（Consistency）
- 新设计是否沿用框架已有的设计模式？
- 命名、接口风格是否与现有代码一致？
- 是否重复造轮子（框架已有类似机制未被利用）？
- **原则**：优先遵循框架约定，除非有充分理由偏离

### 2. 耦合度（Coupling）
- 新模块引入了多少外部依赖？
- 是否产生循环依赖？
- 接口是否足够抽象（依赖接口而非具体实现）？
- **原则**：最小化依赖面，模块间通过明确接口通信

### 3. 扩展性（Extensibility）
- 未来同类需求是否容易在此设计上扩展？
- 是否有预留的扩展点（hook/回调/配置）？
- 数据和配置是否硬编码？
- **原则**：不追求过度设计，但关键扩展点要预留

### 4. 性能（Performance）
- 是否存在明显的性能瓶颈（重复计算、大量内存分配）？
- 热点路径是否足够优化？
- 数据结构的选用是否合理？
- **原则**：先管正确，再管性能；不要过早优化但避开明显坑

### 5. 可测试性（Testability）
- 核心逻辑是否可以脱离环境独立测试？
- 外部依赖是否可以 mock/stub？
- 边界条件是否易于构造？
- **原则**：可测试的代码 = 可维护的代码

### 6. 可读性（Readability）
- 新人把方案看完能否大致理解整体结构？
- 关键概念是否有清晰的命名和注释？
- 复杂度是否控制在合理范围？
- **原则**：代码是给人看的，顺便给机器执行

## 输出格式

```markdown
# 架构评审报告

## 方案概述
[用 2-3 句话描述被评审的方案]

## 逐维度评审

| 维度 | 评分(1-5) | 评价 |
|------|:---------:|------|
| 一致性 | 4 | 重用了框架的 Event 系统，但命名风格与现有模块有差异，见问题 #2 |
| 耦合度 | 3 | ⚠️ 对 DataManager 产生了双向依赖，见问题 #1 |
| 扩展性 | 4 | 预留了配置表，但事件类型硬编码 |
| 性能 | 5 | 无明显性能隐患 |
| 可测试性 | 2 | ❌ 核心逻辑与 UI 层强耦合，无法独立测试，见问题 #3 |
| 可读性 | 4 | 整体清晰，部分函数名可优化 |

**总评: 3.7/5**

## 问题清单

1. **[严重] 双向耦合** — DataManager 也需要导入新模块
   - 影响：增加模块间的维护成本，可能导致循环依赖
   - 建议：通过 Event 通知替代直接调用，DataManager 发布数据变更事件，新模块订阅

2. **[一般] 命名不一致** — 函数命名使用了 get_xxx 而非框架惯用的 xxx_get
   - 影响：降低代码一致性
   - 建议：统一使用 `xxx_get` 模式

3. **[严重] 不可测试** — `CoreLogic.Calculate()` 直接操作 UI 控件
   - 影响：无法在不启动完整 App 的情况下测试计算逻辑
   - 建议：分离计算逻辑和 UI 更新，计算函数返回结果，由调用方更新 UI

## 改进建议（按优先级）

1. 先解耦 UI 和逻辑（问题3）— 这是架构层面的硬伤
2. 用 Event 替代直接调用（问题1）— 降低耦合
3. 统一命名（问题2）— 往后维护成本递减

## 整体结论

□ 通过 — 无需修改
□ 有条件通过 — 修改 [严重] 问题后可执行
☑ 需重新设计 — [1]个严重问题需在编码前解决
```

## 问题分级标准

| 级别 | 定义 | 示例 |
|------|------|------|
| **严重** | 影响架构健康度的结构性问题，必须修正 | 循环依赖、不可测试的架构、违反框架核心约定 |
| **一般** | 影响维护效率，但不影响功能正确性 | 命名不一致、注释缺失、接口不够优雅 |
| **建议** | 锦上添花的优化 | 更好的算法选择、额外的抽象层 |

## 注意事项

- 这是**架构级**评审，不检查代码实现细节（那是品质检查官的活）
- 每个问题必须给出**具体的改进建议**，而非只是批评
- 评分要有依据，不能凭感觉
- 如果方案只有一个维度得分低但其他很好，要说清楚这个维度是不是可容忍的
